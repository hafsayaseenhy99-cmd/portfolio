Fellowship Finder Automation

A daily digest that automatically finds, filters, and delivers postdoctoral fellowships, academic jobs, and grants for Islamic Studies researchers — so they spend time applying, not searching.

Type
Python automation
Stack
Python · SQLite · Gmail API
Sources

The problem

Academic opportunity discovery is broken

Postdoctoral fellowships, funded research positions, and academic grants are scattered across dozens of websites — H-Net, jobs.ac.uk, ACLS, the Mellon Foundation, individual university pages, discipline-specific listservs. There's no single place to look, and the postings that matter most are often buried under irrelevant listings or missed entirely because they appeared on a site you don't check.

For researchers in a specialised field like Islamic Studies, this is especially acute. Generic job boards weren't built for you. You end up doing the same manual trawl every week, on five different sites, for opportunities that may not exist — and occasionally missing one that did.

"The people who most need to know about these opportunities are the ones with the least time to go looking for them."

What I built

A five-component Python pipeline, running daily

The system scrapes, scores, deduplicates, and delivers — fully automated. It runs each morning via a cron job and only emails when there's something genuinely new and relevant.

1
Source aggregation
RSS parsers pull structured feeds from H-Net, jobs.ac.uk, and Times Higher Education. Custom scrapers target FindAPostDoc, ACLS, Mellon, MESA, and others. A Google Custom Search API integration catches smaller sites that don't offer feeds.
sources.py
2
Relevance filtering & scoring
Each result is scored against a keyword taxonomy — exact field matches (islamic studies, quran, hadith, fiqh) score highest; partial matches (MENA, Arabic, Ottoman, theology) score lower; broad matches (religion, Middle East) act as catch-all filters. Expired postings are automatically excluded where dates are detectable.
filters.py
3
Deduplication via SQLite
Every seen opportunity is stored by URL + title hash. Subsequent runs check against this store so the same posting never appears twice, even if it resurfaces across different sources. Once sent, it's marked so it never re-enters the digest.
database.py
4
HTML email digest
New opportunities are grouped by category — Postdocs, Fellowships, Faculty, Grants & Awards, Alt-Academic — and formatted into a clean HTML email showing title, institution, location, deadline, link, and relevance score. Sent via Gmail SMTP using an App Password.
email_digest.py
5
Daily cron orchestration
The main orchestrator runs all scrapers, filters results, stores new items, and sends the digest — only if there's something new. Designed to run silently at 8am daily with a single cron entry.
main.py

Technical design

Deliberately simple. Reliably daily.

The stack is intentionally minimal — no external services, no cloud dependencies, no ongoing costs. Everything runs locally. SQLite handles persistence without a server. Gmail SMTP handles delivery without a mail provider. The only setup a user needs is a Gmail App Password and a cron entry.

The keyword scoring system is designed to be transparent and editable — researchers can open config.py and adjust the keyword lists without touching any scraping logic. That matters when the field you're in has terminology that evolves.

Reflection
What this taught me

This project clarified something I keep coming back to in ops work: the most valuable automation is often the one that removes a recurring, low-value task that nobody has time to eliminate manually. Researchers weren't failing to find fellowships because they weren't smart enough — they were failing because the information infrastructure wasn't designed for them.

The scraping side required more judgement than I expected. Different sites structure their listings differently, update at different cadences, and occasionally change their layouts. Building resilient scrapers meant thinking about failure modes — what happens when a site is down, when a result has no deadline, when a listing resurfaces with a slightly different title. The deduplication logic ended up being one of the most important parts of the system.

The keyword scoring system also taught me something about domain specificity. Islamic Studies isn't just "religion" — the relevant terminology spans Arabic, Persian, and Ottoman historical traditions, specific jurisprudential schools, regional area studies, and contemporary policy contexts. Getting the filter right meant actually understanding the field, not just pattern-matching on obvious terms.
